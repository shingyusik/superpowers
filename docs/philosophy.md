# 하네스 철학

이 문서는 이 포크가 Superpowers에서 출발해 **서브에이전트 중심 하네스**로 진화하기 위한 기본 철학을 정의한다.

구현이 어느 정도 안정될 때까지 문서는 한국어를 기본으로 작성한다. 목적은 빠른 공유, 판단, 수정이다.

## 핵심 방향

Superpowers가 스킬을 통해 메인 에이전트의 행동을 규율한다면, 이 하네스는 메인 에이전트가 여러 서브에이전트의 판단과 실행을 조율하는 구조를 지향한다.

메인 에이전트는 모든 일을 직접 해결하는 존재가 아니다. 목표를 이해하고, 일을 나누고, 적절한 역할에 위임하고, 산출물을 검증하고, 결과를 통합하는 오케스트레이터다. 단, 하네스 개선까지 메인 에이전트가 겸임하지 않는다. 개선은 워크플로우 마지막에 호출되는 전용 하네스 개선 에이전트가 담당한다.

스킬은 이 구조에서 빠지지 않는다. 다만 스킬의 위치가 바뀐다. 스킬은 에이전트를 대신해 판단하는 고정 규칙이 아니라, 각 에이전트가 자기 역할을 수행할 때 참고하는 행동 가이드이자 방법론 정의서다.


## 에이전트 구조

초기 하네스의 에이전트는 크게 세 가지로 나눈다.

- **메인 에이전트**
  - 사용자 목표를 이해하고 전체 흐름을 조율한다.
  - 직접 구현보다 분해, 위임, 통합, 보고에 집중한다.

- **서브에이전트**
  - 메인 에이전트가 위임한 좁은 작업을 수행한다.
  - 서브에이전트 내부는 역할에 따라 더 세분화한다. 초기 역할은 Researcher, Planner, Implementer, Reviewer, Verifier다.

- **하네스 개선 에이전트**
  - 워크플로우 마지막에 항상 호출된다.
  - 전체 작업 기록을 검토해 스킬, 에이전트, 산출물 템플릿, task contract 템플릿 개선을 담당한다.

이 구조에서 서브에이전트의 세부 역할은 필요에 따라 늘리거나 줄일 수 있다. 하지만 최상위 분류는 먼저 세 가지로 유지한다.

## 핵심 철학

- **Delegation-first**
  - 메인 에이전트의 직접 해결보다 적절한 분해와 위임을 우선한다.
  - 단, 위임 자체가 목적은 아니다. 작고 명확하게 맡길 수 있을 때 위임한다.

- **Orchestrated over solo**
  - 단일 에이전트의 긴 사고보다 역할이 분리된 협업 흐름을 선호한다.
  - 메인 에이전트의 핵심 가치는 구현량이 아니라 조율 품질이다.

- **Isolated context**
  - 서브에이전트는 필요한 맥락만 받는다.
  - 좁은 맥락은 판단을 선명하게 하고, 불필요한 오염을 줄인다.

- **Independent verification**
  - 산출물을 만든 에이전트와 검증하는 흐름을 분리한다.
  - 서브에이전트의 성공 보고는 증거가 아니라 검증 대상이다.

- **Context reduction**
  - 복잡도는 코드뿐 아니라 컨텍스트에도 존재한다.
  - 하네스는 각 에이전트가 들고 있는 맥락의 크기와 책임 범위를 줄여야 한다.

- **Cross-checked evidence**
  - 주장은 테스트, diff, 로그, 재현 절차, 독립 리뷰로 확인되어야 한다.
  - 하나의 보고보다 교차 확인된 증거를 신뢰한다.

- **Progressive evolution**
  - 하네스는 처음부터 완성된 규칙 체계가 아니다.
  - 실제 사용, 실패, 반복되는 패턴을 통해 점진적으로 구조를 조정한다.
  - 모든 워크플로우 마지막에는 개선 에이전트를 호출해 작업 기록 전체를 검토하고 구조 개선 후보를 찾는다.

- **Loose rules, strong boundaries**
  - 세부 행동은 에이전트가 판단하게 둔다.
  - 대신 책임 범위, 금지 범위, 검증 기준, 승인 경계는 명확히 둔다.

- **Skills as methodology**
  - 스킬은 각 에이전트의 행동 가이드이자 방법론 정의서다.
  - 에이전트는 스킬을 기계적으로 따르는 실행기가 아니라, 스킬을 참고해 상황에 맞게 판단하고 실행하는 주체다.

## Superpowers에서 계승하는 것

- 테스트와 검증을 중시한다.
- 즉흥적 수정보다 체계적 절차를 선호한다.
- 불필요한 복잡도를 줄인다.
- 증거 없이 성공을 주장하지 않는다.

## Superpowers에서 바꾸는 것

- 스킬을 제거하지 않고, 스킬의 역할을 재배치한다.
- 스킬은 중앙 통제 규칙보다 역할별 방법론 문서에 가까워진다.
- 메인 에이전트 규율보다 서브에이전트 역할 분리를 우선한다.
- 하네스 개선 책임을 메인 에이전트에서 분리해 전용 하네스 개선 에이전트에 둔다.
- 단일 실행 흐름보다 위임, 검증, 통합 루프를 명시한다.
- 고정 규칙보다 점진적으로 진화하는 운영 구조를 선호한다.

## 설계 판단 기준

새 기능이나 문서를 추가할 때 다음 질문으로 판단한다.

- 이 변경은 위임을 더 명확하게 만드는가?
- 필요한 스킬이 어떤 에이전트의 어떤 판단을 돕는지 분명한가?
- 서브에이전트의 책임 범위를 줄이는가?
- 검증 가능성을 높이는가?
- 메인 에이전트의 조율 판단을 돕는가?
- 개선 에이전트가 작업 기록을 보고 판단할 수 있는 증거가 남는가?
- 규칙을 늘리는 대신 경계를 명확하게 하는가?
- 실제 실패나 반복 패턴에서 나온 변화인가?


## 개선 에이전트

하네스 개선은 메인 에이전트의 부가 업무가 아니라 별도 역할이다.

개선 에이전트는 모든 워크플로우 마지막에 호출된다. 이 에이전트는 전체 작업 기록, task contract, 산출물, 검증 결과, 리뷰 결과, 사용자 피드백을 살펴보고 다음 개선 대상을 찾는다.

개선 대상:

- 스킬: 행동 가이드와 방법론 정의서
- 에이전트: 역할, 책임, 입력/출력
- 산출물 템플릿: 결과 보고, 리뷰, 검증 보고 형식
- task contract 템플릿: Goal, Context, Boundaries, Expected output, Verification, Escalation

개선 에이전트는 직접 구조를 무제한 변경하지 않는다. 반복 가능성이 있는 문제와 개선안을 식별하고, 필요한 경우 문서나 템플릿 변경 제안을 남긴다.

## 현재 단계의 원칙

초기에는 완성된 런타임보다 문서, 계약, 템플릿, 역할 정의를 먼저 만든다.

구현은 문서화된 루프를 반복 사용해 본 뒤 진행한다. 먼저 구조를 고정하지 않는다. 작게 만들고, 써보고, 고친다.
